December 8, 2025 · One min read
  1. 일단 API 관련해서

APP_URL, API_URL, NEXT_API_URL

API_URL 은 서버측 코드에서 사용함. 실제 데이터 베이스나 다른 백엔드 서버와 통신할 떄 사용. NEXT_API_URL 은 클라이언트 측에서 서버 컴포넌트나 API route가 내부적으로 호출하는 내부 URL, 백엔드 서버 URL.

  1. 로그인 관련 시스템

  • 로그인 관련 시스템에서 정말 에러 많이 발생함. 로그인 관련해서 이해한 것들을 간단하게 정리해보자.

  • 일단 클라이언트가 서버로 로그인 요청을 보냄 : NEXT_API_URL 사용. api.xxxx.com

  • 서버는 클라이언트에 쿠키를 생성하고, 쿠키를 response 로 클라이언트에 전송함. 이때 서버에서 받아온 쿠키의 도메인 정보는 api.xxx.com

  • 클라이언트는 해당 정보를 저장.

  • 이후 클라이언트가 인증 정보가 필요한 페이지에 접속할때 서버에 쿠키 정보를 포함해서 요청을 보냄. api.xxx.com

  • 대부분 클라이언트는 도메인 주소를 확인하기 떄문에 api.xxx.com 과 xxx.com 이 다르다고 인지한다. 이 때문에 쿠키 허용 url을 .xxx.com 으로 바꿔준다.

nginx 를 통해서 리버스 프록시를 구축.

RAG & Milvus 간단 정리

November 24, 2025 · 3 min read

RAG

  • 데이터 indexing 체계가 더 중요하다고 생각됨.
  • pre-retrieval, retrieval-time, post-retrieval 검사가 전부 이루어짐.

Faiss Index

  • ANN 기반
  • IndexHNSWFlat : 빌드 O(NlogN) 백만개에 수초. 서치 O(logN) 백만개 문서 기준 밀리초.
  • Hierarchical Navigable Small World
  • 계층적 그래프를 그리고 최단 거리 기반으로 탐색. 계층을 타고 목표 지점 근처로 빠르게 간 후, 정밀 탐색하는 방식.

Milvus

  • faiss 를 핵심으로 하는 벡터DB - DB의 테이블 역할.
  • Milvus에서 벡터를 저장하는 논리적 단위 : "컬렉션"
  • ANN 구조를 활용해서 벡터를 저장
  • Faiss/ milvus 모두 병렬 처리 환경에 최적화 되어 있음.

Milvus 구축예시 : 테이블 역할을 하는 컬렉션 생성.

from pymilvus import connections, utility, Collection, FieldSchema, CollectionSchema, DataType
import numpy as np

# 1. Milvus 서버 연결 설정
# 로컬에서 실행 중인 Milvus 서버에 연결합니다. (기본 포트: 19530)
# Milvus 서버가 다른 곳에 있다면 주소를 변경해야 합니다.
try:
    connections.connect(alias="default", host="localhost", port="19530")
    print("✅ Milvus 서버 연결 성공.")
except Exception as e:
    print(f"❌ Milvus 서버 연결 실패: {e}. 서버가 실행 중인지 확인하세요.")
    exit()

COLLECTION_NAME = "rag_document_chunks"
DIMENSION = 128  # 벡터 차원 (예: 임베딩 모델의 출력 차원)

# 컬렉션이 이미 존재하면 삭제합니다.
if utility.has_collection(COLLECTION_NAME):
    utility.drop_collection(COLLECTION_NAME)
    print(f"기존 컬렉션 '{COLLECTION_NAME}' 삭제 완료.")

# 2. 컬렉션 스키마 정의 (데이터 구조 정의)
# Milvus는 벡터 외에 메타데이터도 저장해야 합니다.

fields = [
    # 필드 1: primary key (고유 ID)
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True, auto_id=True, description="Primary ID"),
    
    # 필드 2: RAG의 원본 텍스트 청크를 식별하는 ID (메타데이터)
    FieldSchema(name="chunk_id", dtype=DataType.VARCHAR, max_length=256, description="Original Chunk ID"),
    
    # 필드 3: 벡터 필드 (가장 중요)
    FieldSchema(name="embedding", dtype=DataType.FLOAT_VECTOR, dim=DIMENSION, description="Vector embedding")
]

schema = CollectionSchema(fields, description="RAG Document Chunks Collection")

# 3. 컬렉션 생성
collection = Collection(name=COLLECTION_NAME, schema=schema)
print(f"📦 컬렉션 '{COLLECTION_NAME}' 생성 완료.")

Milvus 데이터 삽입: 벡터 insert -> index 구축 순서로 진행됨.

# --- 4. 데이터 준비 (RAG 임베딩 시뮬레이션) ---
num_entities = 1000  # 삽입할 벡터의 수
rng = np.random.default_rng(1234)

# 128차원의 랜덤 벡터 1000개 생성
data = [rng.random((DIMENSION,)).astype('float32') for _ in range(num_entities)]
# 메타데이터 (RAG에서 실제 텍스트 청크와 연결될 ID)
chunk_ids = [f"chunk_{i}" for i in range(num_entities)]

# 데이터 구조화 (Milvus 삽입 형식)
entities = [
    chunk_ids,
    data
]

# --- 5. 데이터 삽입 ---
insert_result = collection.insert(entities)
print(f"📥 {len(insert_result.primary_keys)}개의 엔티티 삽입 완료.")

# --- 6. 인덱스 구축 (핵심 RAG 단계) ---
# Milvus에게 Faiss의 HNSW 알고리즘을 사용하여 인덱스를 만들도록 지시합니다.
# 이 인덱스는 실제로 Milvus 내에서 Faiss 엔진을 사용합니다.
index_params = {
    "index_type": "HNSW",
    "metric_type": "L2",  # 유클리드 거리 (L2) 사용
    "params": {"M": 16, "efConstruction": 200} # HNSW 하이퍼파라미터
}

collection.create_index("embedding", index_params)
print("⚙️ 인덱스 HNSW 구축 완료.")

# 데이터를 검색에 사용할 수 있도록 메모리에 로드합니다.
collection.load() 
print("메모리 로드 완료.")

벡터 검색:

# --- 7. 질의 벡터 준비 (사용자 질문 임베딩 시뮬레이션) ---
# 단일 질의 벡터 생성
query_vector = rng.random((1, DIMENSION)).astype('float32')

# --- 8. 검색 수행 ---
TOP_K = 5 # 가장 가까운 5개 벡터를 검색

search_params = {
    "metric_type": "L2",
    "params": {"ef": 50} # HNSW 검색 매개변수 (efSearch와 유사)
}

start_time = time.time()
results = collection.search(
    data=query_vector,       # 검색할 질의 벡터
    anns_field="embedding",  # 검색을 수행할 필드 (벡터 필드)
    param=search_params,     # 검색 매개변수
    limit=TOP_K,             # Top-K
    output_fields=["chunk_id"] # 결과로 반환받을 메타데이터 필드
)
end_time = time.time()

print(f"\n🔍 검색 완료. (소요 시간: {end_time - start_time:.4f}초)")

# --- 9. 결과 출력 ---
print(f"\n✅ 검색 결과 (Top-{TOP_K}):")
for hit in results[0]:
    # hit.id는 Milvus의 내부 ID, hit.entity.get('chunk_id')는 메타데이터
    print(f"  - Chunk ID: {hit.entity.get('chunk_id')}, 유사도(거리): {hit.distance:.4f}")

# 작업 후 연결 종료
connections.disconnect("default")

첫 서비스 배포기

November 12, 2025 · 2 min read

내 서버를 만들고 서비스를 하나 배포했다. AWS 같은 걸 고민했는데 집에 굴러다니는 오래된 컴퓨터를 포맷하고 우분투를 깔았다.

언제 사라질지 모르는 링크. 다양한 플랫폼 요약을 자동으로 모아서 업데이트해주는 것에 의미를 뒀다.

https://vince-convenable-sinistrorsely.ngrok-free.dev/content_viewer

맥 개발 환경이었기떄문에 그렇게 큰 문제 없이 배포가 가능할줄알았는데, 역시나 문제가 많이 발생해서 꽤 고생했다. 이틀정도 작업 시간을 여기 많이 투자한 듯.

개발환경과 배포환경의 완전한 일치는 인류의 염원으로만 남아있을까?

서비스 환경에 배포는 무조건 도커, 쿠버네티스를 추구한다. 가벼운 앱이라 고민하고 도커 배포로 진행했다.

내 개발 철학을 되돌아보고 다듬어볼 수 있었다.

항상 목표를 크게 세우는 편인데 이번 개발은 0.0.1, 0.0.2 에서 목표를 가볍게 잡고 각 과정을 수행해서 빠르게 끝낼 수 있었다.

개발 과정에서 가장 중시한 건 목적중심, 가변성, 속도였다.

가장 중요한 건 목표다. 목표를 명확하고 짧게 잡았다.

두 번쨰로는 이 코드의 기능을 수정하거나 대체하기 편한가. 외부 환경이 바뀌어도 이 코드는 큰 수정 없이 사용 가능한가. 그래서 대부분의 기능을 layer를 분리하고 layer 와 layer 는 서로의 내부 상태나 구현은 신경 쓸 필요없게 구현했다.

마지막이 속도다. 1,2 요건이 충족되는 한 무조건 가장 빠르게 구현해볼 수 있는 방식을 선택했다. 예를 들어서 초기 개발 단계에서는 DB연결 레이어를 설계한 뒤 DB는 일단 로컬 duckdb를 사용해서 친구들에게 1차로 공개했다. 이 후 확장성을 고려해서 pg 로 연결했으나 이 과정에서 나는 다른 부분은 전혀 건드리지 않고 DB - DB 레이어 연결만 수정하면 됐다.
